# RAG Against the Machine

## Description

This project implements a Retrieval-Augmented Generation system for answering questions about the vLLM repository.

The goal of the project is to build a complete RAG pipeline able to:

- ingest the vLLM repository;
- process Python source code and Markdown/text documentation;
- split files into searchable chunks;
- build a persistent lexical index;
- retrieve relevant sources for a question;
- generate answers with a local LLM using the retrieved context;
- output structured JSON compatible with the provided evaluation tools.

The default language model used for answer generation is:

```text
Qwen/Qwen3-0.6B
```

The retrieval backend is based on TF-IDF and cosine similarity. The system builds separate indices for documentation and code, plus a combined index for general interactive questions.

## System Architecture

The system is organized as a classic RAG pipeline:

```text
Raw vLLM repository
        |
        v
File discovery
        |
        v
Chunking
        |
        v
Chunk storage
        |
        v
TF-IDF index creation
        |
        v
Retrieval
        |
        v
Prompt construction
        |
        v
LLM answer generation
        |
        v
Structured JSON output
```

### Components

The main components are:

- `src/cli.py`: command-line interface built with Python Fire.
- `src/cli_models.py`: Pydantic models for CLI argument validation.
- `src/models.py`: Pydantic models for chunks, datasets, search results, and answers.
- `src/markdown.py`: Markdown and text file processing.
- `src/python.py`: Python source file processing.
- `src/chunking.py`: generic chunk splitting and validation helpers.
- `src/storage.py`: chunk serialization and loading.
- `src/tfidf.py`: TF-IDF index building and ranked retrieval.
- `src/retrieval.py`: source retrieval, context construction, and prompt construction.
- `src/cli_execution.py`: execution logic for indexing, searching, and answering.

### Stored outputs

The indexing step creates chunk files:

```text
data/processed/chunks/documents.json
data/processed/chunks/code.json
```

It also creates three TF-IDF indices:

```text
data/processed/tfid_documents_index/
data/processed/tfid_code_index/
data/processed/tfid_all_index/
```

The three indices are used as follows:

```text
documents index -> documentation dataset questions
code index      -> code dataset questions
all index       -> general interactive search and answer commands
```

## Chunking Strategy

The project implements two chunking strategies:

1. Markdown/text chunking;
2. Python code chunking.

Both strategies preserve source traceability with:

```text
file_path
first_character_index
last_character_index
content
```

This allows each retrieved result to point back to the original file and character range.

### Markdown and text chunking

Markdown and text files are read recursively from:

```text
data/raw/
```

The documentation chunker processes:

```text
.md
.txt
```

It ignores directories that should not be indexed, such as:

```text
.git
__pycache__
.venv
venv
node_modules
dist
build
.mypy_cache
.pytest_cache
.ruff_cache
```

The Markdown strategy first splits documents around top-level title boundaries. This keeps chunks close to natural documentation sections.

If a section is larger than the configured maximum chunk size, it is split again using the generic chunking helper.

### Python code chunking

Python files are also read recursively from:

```text
data/raw/
```

The code chunker processes:

```text
.py
```

Each Python file is initially treated as a source unit. If the file is larger than the configured maximum chunk size, it is split into smaller chunks.

This approach is simple and robust. It preserves character offsets, which are required by the evaluation format.

### Generic chunking

The generic chunking helper supports:

- configurable maximum chunk size;
- configurable overlap;
- offset-aware chunk creation;
- validation of character ranges.

The evaluation command uses:

```bash
uv run python -m src index --max_chunk_size 2000
```

### Chunk size trade-off

If chunks are too small:

- relevant context may be fragmented;
- the answer may require several chunks to become understandable;
- retrieval may return incomplete evidence.

If chunks are too large:

- retrieved chunks may contain too much unrelated information;
- retrieval precision may decrease;
- the LLM receives noisier context;
- fewer useful chunks fit in the prompt.

The implemented strategy uses moderately sized chunks to balance context preservation and retrieval precision.

## Retrieval Method

The retrieval system uses TF-IDF with cosine similarity.

TF-IDF means Term Frequency-Inverse Document Frequency.

The idea is:

- words that appear often in a chunk are important for that chunk;
- words that appear in many chunks are less discriminative;
- words that appear in fewer chunks help identify more specific sources.

The retrieval process is:

```text
1. Build a corpus from the indexed chunks.
2. Fit a TF-IDF vectorizer on that corpus.
3. Store the vectorizer and TF-IDF matrix on disk.
4. Transform the user query using the same vectorizer.
5. Compute cosine similarity between the query vector and every chunk vector.
6. Sort chunks by descending similarity score.
7. Return the top-k chunks.
```

The project uses:

```text
TfidfVectorizer
cosine_similarity
joblib
```

The corpus includes both:

- chunk content;
- normalized file path information.

Including file path information improves retrieval for code questions, because function names, module names, and directory names can be relevant.

The text normalizer also improves matching for code-like terms by splitting:

```text
/
_
.
-
camelCase
PascalCase
```

This helps natural language queries match source code identifiers.

## Design Decisions

### TF-IDF instead of embeddings

TF-IDF was chosen because it is:

- simple;
- deterministic;
- explainable;
- fast to index;
- fast to query;
- sufficient to pass the required retrieval evaluation.

It also avoids the complexity of downloading or serving embedding models.

### Separate documentation and code indices

Documentation and code have different structures.

Documentation is mostly natural language.

Code contains:

- identifiers;
- function names;
- class names;
- import paths;
- file paths;
- implementation details.

Using separate indices for dataset evaluation improves retrieval focus:

```text
dataset_docs_* -> documentation index
dataset_code_* -> code index
```

### Combined index for interactive commands

The project also builds a combined index.

Manual commands such as:

```bash
uv run python -m src search "How to configure OpenAI server?" --k 10
```

and:

```bash
uv run python -m src answer "How to configure OpenAI server?" --k 10
```

use the combined index by default.

This is useful because a user may not know whether the answer is in documentation, code, or both.

### Character offsets

Each retrieved source includes character offsets:

```text
first_character_index
last_character_index
```

This makes the result auditable and compatible with the evaluation format.

### Deterministic answer generation

The answer command uses deterministic generation:

```text
do_sample=False
```

This reduces output variability between runs.

## Challenges Faced

### 1. Matching natural language questions to code

Code uses identifiers such as function names, class names, and file paths. User questions are usually written in natural language.

Solution:

- include file path information in the TF-IDF corpus;
- normalize separators such as `/`, `_`, `.`, and `-`;
- split mixed-case identifiers.

### 2. Supporting both documentation and code

Documentation and source code behave differently during retrieval.

Solution:

- create one documentation index;
- create one code index;
- create one combined index.

Dataset evaluation uses the specialized indices, while manual commands use the combined index.

### 3. Keeping source references valid

The evaluator expects file paths and character ranges.

Solution:

- every chunk stores its original file path;
- every chunk stores zero-based character offsets;
- retrieved sources are created directly from chunk metadata.

### 4. Handling long files

Large files may exceed the maximum chunk size.

Solution:

- use a generic chunking helper;
- split large sections while preserving offsets;
- support overlap between chunks.

### 5. Avoiding uncontrolled crashes

The evaluation includes edge cases such as empty queries and missing dataset paths.

Solution:

- validate CLI arguments with Pydantic;
- catch common errors;
- print controlled error messages to standard error.

All external data models and validation models use Pydantic. TfidfRetriever is not a data model; it is a service class that implements retrieval logic. ChunkValidationError is an enum. SearchResult is an internal ranked result object.
